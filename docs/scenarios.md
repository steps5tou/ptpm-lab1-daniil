# Пользовательские сценарии модуля «Сервис задач»

## Роли

Пользователь — студент, который создаёт задачи и управляет ими.
Система — модуль «Сервис задач», который обрабатывает запросы и хранит данные.

## Сценарии

### S-01. Создание задачи
Приоритет: критический

Цель: пользователь хочет создать новую задачу.
Условия: файл данных доступен или отсутствует.
Шаги:
1. Пользователь передаёт название и описание задачи.
2. Система проверяет, что название не пустое.
3. Система назначает задаче уникальный идентификатор и статус «новая».
4. Система сохраняет задачу в файл JSON.
Ожидаемый результат: задача создана, получила идентификатор, появилась в списке, сохранена в файле.

---

### S-02. Просмотр списка задач
Приоритет: критический

Цель: пользователь хочет увидеть все свои задачи.
Условия: файл данных доступен, задачи существуют.
Шаги:
1. Пользователь запрашивает список задач.
2. Система загружает задачи из файла JSON.
3. Система возвращает полный список задач.
Ожидаемый результат: пользователь получает список всех задач с полями id, title, description, status.

---

### S-03. Изменение статуса задачи
Приоритет: критический

Цель: пользователь хочет отметить задачу как выполненную.
Условия: задача с указанным идентификатором существует.
Шаги:
1. Пользователь передаёт идентификатор задачи и новый статус.
2. Система проверяет, что задача существует.
3. Система проверяет, что новый статус допустим.
4. Система обновляет статус и сохраняет изменения в файл.
Ожидаемый результат: статус задачи изменён, изменение сохранено в файле.

---

### S-04. Поиск задачи по ключевому слову
Приоритет: важный

Цель: пользователь хочет найти задачу по слову в названии или описании.
Условия: задачи существуют.
Шаги:
1. Пользователь передаёт ключевое слово для поиска.
2. Система выполняет поиск без учёта регистра по полям title и description.
3. Система возвращает список совпадений.
Ожидаемый результат: пользователь получает список задач, содержащих ключевое слово.

---

### S-05. Загрузка задач из файла при старте
Приоритет: критический

Цель: система восстанавливает задачи после перезапуска.
Условия: файл JSON существует и не повреждён.
Шаги:
1. Система инициализируется и обнаруживает файл tasks.json.
2. Система читает файл и загружает задачи в память.
Ожидаемый результат: все ранее сохранённые задачи доступны для работы.

---

### S-06. Отсутствие файла данных при старте
Приоритет: критический

Цель: система корректно запускается без файла данных.
Условия: файл tasks.json отсутствует.
Шаги:
1. Система инициализируется и не обнаруживает файл tasks.json.
2. Система начинает работу с пустым списком задач.
Ожидаемый результат: система запущена без ошибок, список задач пуст, файл будет создан при первом сохранении.

---

### S-07. Повреждённый файл данных при старте
Приоритет: важный

Цель: система сообщает об ошибке при повреждённом файле.
Условия: файл tasks.json существует, но содержит некорректные данные.
Шаги:
1. Система инициализируется и обнаруживает файл tasks.json.
2. Система пытается прочитать файл и получает ошибку разбора.
3. Система возвращает сообщение о повреждении файла.
Ожидаемый результат: система не перезаписывает файл автоматически, пользователь получает понятное сообщение об ошибке.

---

### S-08. Попытка установить недопустимый статус
Приоритет: важный

Цель: система отклоняет недопустимое значение статуса.
Условия: задача с указанным идентификатором существует.
Шаги:
1. Пользователь передаёт идентификатор задачи и недопустимый статус.
2. Система проверяет значение статуса.
3. Система обнаруживает, что статус не входит в список допустимых.
4. Система возвращает ошибку с указанием допустимых значений.
Ожидаемый результат: статус задачи не изменён, пользователь получает сообщение об ошибке.

---

### S-09. Создание задачи с пустым названием
Приоритет: важный

Цель: система отклоняет создание задачи без названия.
Условия: пользователь передаёт пустую строку как название.
Шаги:
1. Пользователь передаёт пустое название задачи.
2. Система проверяет название.
3. Система обнаруживает, что название пустое.
4. Система возвращает ошибку о недопустимом значении.
Ожидаемый результат: задача не создана, файл не изменён.

---

### S-10. Фильтрация списка по статусу
Приоритет: дополнительный

Цель: пользователь хочет увидеть только задачи с определённым статусом.
Условия: задачи с разными статусами существуют.
Шаги:
1. Пользователь запрашивает список задач с фильтром по статусу.
2. Система возвращает только задачи с указанным статусом.
Ожидаемый результат: список содержит только задачи с нужным статусом.

---

## Проверяемость

### S-01. Создание задачи
1. Что будет, если передать пустое название — вернётся ли ошибка или задача всё равно создастся?
2. Как убедиться, что сохранение прошло — появится ли задача в файле tasks.json сразу после создания?

### S-08. Попытка установить недопустимый статус
1. Что будет, если передать статус на другом языке или с опечаткой — изменится ли статус задачи?
2. Как убедиться, что текущий статус не изменился после отклонения — вернёт ли система прежнее значение?
